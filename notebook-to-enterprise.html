Engineering structure

FAIR-VID uses notebooks as the source layer for later Python services.

This is a core FAIR-VID claim in the paper. The framework is prototyped as executable notebooks, then migrated or synchronized into server-side Python with prompt templates that preserve inspectable outputs and local execution constraints.

Framework flow

How paper, notebooks, prompts, and scripts fit together

1

Paper defines the architectural pattern

The manuscript argues for modular notebook agents, symmetric transparency, and interpretable orchestration. This is the conceptual layer.

2

Notebooks become the executable specification

Each agent is built and tested in Colab with visible intermediate artifacts and shared storage for framework users, code users, and paper readers.

3

Prompt templates migrate or synchronize logic

FAIR-VID uses prompt templates to either create a first server script from a notebook or update an existing server script after the notebook changes.

4

Server-side Python preserves inspectability

Local scripts keep relative paths, log files, JSON outputs, prompt traceability, and applicant-folder organization so the deployment step does not destroy inspectability.

Concrete examples

Examples already present in FAIR-VID

  • Document scripts use applicant-application subdirectories and prompt folders from the FAIR-VID prompt repository.
  • Transcript grading scripts fetch interview-evaluator prompts and save one JSON per transcript.
  • Process-mining code preserves the notebook’s missing-ID recovery, event normalization, and simulation logic.
System implication

The method covers IS implementation and model evaluation

FAIR-VID specifies AI systems in high-stakes domains as versionable, testable components. The prompt-guided notebook-to-code bridge is part of that method because it maps prototype logic to institutional infrastructure.
