Notebook-to-Enterprise Development Paradigm

A core contribution of the FAIR-VID framework is the reproducible method for developing AI-infused information systems. Rather than building monolithic solutions, the system is first prototyped as a collection of modular, transparent Jupyter notebooks. These components are subsequently transformed into enterprise-ready server scripts using AI programming agents.

1. Methodological Approach

In the proposed digital twin architecture, transparency is enforced by separating development and inspection from execution. The framework establishes a unified deployment methodology:

  • Prototyping Layer (Colab Notebooks): Used for experimental development, community inspection, and step-by-step auditing. Intermediate artifacts (e.g., OCR bounding boxes, JSON scores) are highly visible.
  • Execution Layer (Server-side Python): AI programming agents (such as GPT-4o or GPT-5.3-Codex) are used to convert the notebooks into local Linux scripts. These scripts preserve process logic, remove cloud dependencies (like Google Drive mounts), and utilize local models via tools like Ollama.

2. The Prompt Repository for Code Generation

To standardize the IS development transition from prototype to enterprise, we have formalized the prompt templates used by the AI programming agents. The underlying design principle governing these templates is:

Design Principle: "Colab notebooks act as the transparent source layer, while server-side Python scripts act as the production-oriented execution layer. The transition must preserve local paths, Ollama execution, JSON output validation, and state-centric process mapping."

These formalized instructions are hosted in a dedicated, open-source repository.

Adapter Prompt (Initial Generation)

Used when translating a newly validated Colab notebook into its first local Python executable. The agent is instructed to isolate the business logic from notebook-specific artifacts.

Target constraints defined in prompt:
  • Remove !pip install and /content/drive paths.
  • Follow local Ollama communication patterns.
  • Construct standalone if __name__ == "__main__": blocks.
File: notebook_to_script_adapter_prompt.md "Replicate all logic from the notebook. Follow the Ollama integration pattern, including API calls, model selection, prompt formatting, and JSON parsing. Use relative paths instead, assuming the script runs in a directory where dream_applicant_application/ is a direct subdirectory."

Update Prompt (Incremental Synchronization)

Used when the process-centric rules or evidence-extraction parameters change in the notebook, and the existing server script must be updated without overwriting the established local infrastructure.

Target constraints defined in prompt:
  • Do not recreate the script from scratch.
  • Compare the new logic and implement targeted changes.
  • Preserve existing error handling and local directory structures.
File: colab-to-server-script-update-prompt.md "Compare the new Colab notebook with the existing server script and modify the server script so it implements all new logic while preserving local Linux execution, relative paths, JSON handling, error handling, and standalone main() structure."

3. Applied Examples within the Framework

The notebook-to-enterprise methodology was rigorously applied across the three primary branches of the FAIR-VID architecture to process the 18,434 historical applications discussed in the paper:

Process Mining Automation

Prototype: Event logs from DreamApply were analyzed in a notebook to handle non-linear workflow paths (the "spaghetti" effect).
Enterprise State: Mapped via the update prompt into Journal_events_analysis.py, establishing the state-centric orchestration script that loops through institutional logs locally.

Multimodal Document extraction

Prototype: Document completeness and Gemma-3 visual understanding logic developed interactively.
Enterprise State: Converted via the adapter prompt into doc_executiveSummary.py, scaling the pipeline to iteratively process local applicant_application subdirectories without relying on cloud storage.

Video Transcript Evaluation

Prototype: Whisper transcription and LLM-based rubric grading configured in Colab.
Enterprise State: Standardized into video_interview_transcript_grade_ollama.py. The programming agent successfully mapped the transcript evaluation workflow to execute entirely via local Ollama calls, ensuring regulatory data protection.
