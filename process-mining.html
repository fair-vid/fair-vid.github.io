Process Mining & State-Centric Orchestration

This page details the formal data abstraction pipeline and mathematical formulation used to compute the process replay fidelity metrics presented in the paper (Table 1). It provides a direct link between the academic definitions and the provided executable Colab simulation using synthetic data.

1. Formal Pipeline Architecture

Raw event logs suffer from a "granularity gap," where human administrators execute multiple system clicks to complete a single high-level business task. The simulation code bridges this gap through a 4-step formalization:

Step 1: Time-Based Event Abstraction (Sessionization)

Let an application trace be a chronological sequence of events \( E = \langle e_1, e_2, \dots, e_n \rangle \) with timestamps \( T = \langle t_1, t_2, \dots, t_n \rangle \). A new working session \( S \) is triggered if the time elapsed between consecutive events exceeds a predefined threshold \( \tau \) (e.g., 30 minutes):

\( \Delta t = t_i - t_{i-1} > \tau \)
Admission Context Example An administrator clicks "View Profile", then 2 minutes later clicks "Open Transcript", and 1 minute later clicks "Verify Education". Because the time between these events is strictly less than the \(\tau = 30\) minute threshold, the algorithm clusters them into a single logical session (\(S_1\)) rather than treating them as three independent process steps.

Step 2: Domain-Driven Event Aggregation (Head Actions)

To determine the primary business outcome of session \( S_k \), we introduce a domain-defined weight function \( W(e) \in [0,1] \) representing the priority of an event. The abstracted "Head Action" \( h_k \) for session \( S_k \) is:

\( h_k = \arg\max_{e \in S_k} W(e) \)
Admission Context Example Within session \(S_1\), the institutional policy dictionary assigns a high business weight to "Verify Education" (\(W=0.9\)) and a low weight to mere navigation like "View Profile" (\(W=0.1\)). The algorithm selects "Verify Education" as the Head Action (\(h_1\)), successfully filtering out the system's navigational noise.

Step 3: Trace Prefix Encoding (State History Construction)

To accurately predict the next action while preserving context, the model requires the historical trace prefix. For any step \( k \), the "State" \( \Sigma_k \) is the cumulative trace up to that point (\( \oplus \) denotes concatenation):

\( \Sigma_k = \langle h_1 \oplus h_2 \oplus \dots \oplus h_k \rangle \)
Admission Context Example If an application is at step 3, the algorithm does not just look at the current status ("Interview Graded"). It mathematically encodes the exact sequence taken: \(\Sigma_3 = \text{"Application Submitted"} \rightarrow \text{"Education Verified"} \rightarrow \text{"Interview Graded"}\). This ensures the model respects the exact procedural path the applicant went through.

Step 4: Next-Activity Prediction & Data-Aware Mining

In the purely structural model, a Decision Tree \( \mathcal{T} \) approximates \( f(\Sigma_k) \approx h_{k+1} \). In the Data-Aware execution (integrating domain attributes), a feature vector \( A_i = \langle f_1, f_2, \dots, f_m \rangle \) of the applicant is merged with the state history:

\( X_{i,k} = [ \Sigma_{i,k}, f_{i,1}, f_{i,2}, \dots, f_{i,m} ] \)
Admission Context Example To predict the human administrator's next move, the Decision Tree evaluates the combined payload: \(X = [\Sigma_3, \text{AI_Doc_Score}: 95, \text{AI_Interview_Score}: 82]\). It dynamically learns from historical data that an applicant with this specific process history and an interview score of 82 should be routed to the "Issue Conditional Offer" state.

2. Interpreting Process Replay Fidelity (Table 1)

Section 6 of the paper presents the evaluation of the digital twin via replay simulation. Below is the exact definition of the metrics used to populate Table 1, computed for \( N \) total application traces. For a trace \( i \), \( L_i \) denotes its total steps, \( y_{i,j} \) the historical action, and \( \hat{y}_{i,j} \) the predicted action.

Paper reference: Table 1 (Excerpt)
Metric Description Value
Completion rate Simulated cases reaching a final state 99.2%
Action agreement Average step-by-step match per trace 87.5%
High-sim. traces Cases with trace similarity ≥ 0.85 61.4%
Divergence step First deviating step (mean) 3.4
Outcome agreement Agreement with final historical action 87.3%

Mathematical Formulation

Trace Similarity Score (\( S_i \)):
\( S_i = \frac{1}{L_i} \sum_{j=1}^{L_i} \mathbb{I}(\hat{y}_{i, j} = y_{i, j}) \)

Action Agreement: Arithmetic mean of \( S_i \) across all \( N \) traces.
\( (\frac{1}{N} \sum_{i=1}^{N} S_i) \times 100\% \)

High-Sim Traces:
\( (\frac{1}{N} \sum_{i=1}^{N} \mathbb{I}(S_i \ge 0.85)) \times 100\% \)

Divergence Step (\( D_i \)): Lowest index where mismatch occurs.
\( D_i = \min \{ j \mid \hat{y}_{i, j} \neq y_{i, j} \} \)

3. Executable Demonstration

To allow readers to verify the logic and recreate Table 1 without exposing institutional PII, we provide a synthetic data implementation hosted via Google Colab.

Process_Mining_Simulation.ipynb

This executable notebook implements the 4-step algorithm defined above. It pulls a synthetic TSV event log (df_sim_source_simple_synthetic.tsv), handles time-based sessionization, encodes sequence history, and trains the white-box decision trees to calculate Replay Fidelity.

Run Demo in Google Colab

Note on Synthetic Data

The public demo runs on synthetically generated log structures to protect privacy. These logs replicate the structural properties and schema of the original system (timestamps, action sets, payload parameters) but do not represent actual student data. The exact evaluation code from the paper is preserved.
