Process-mined foundation

The FAIR-VID digital twin starts from historical workflow traces, not an idealized process chart.

Process mining is the control layer of FAIR-VID. Historical event logs are cleaned, linked, normalized, mined, transformed into explicit states, and then replayed with next-activity prediction logic. This defines the transition from process discovery to process orchestration.

Notebook chain

Notebooks used in this flow

Journal_events_analysis.ipynb prepares and mines the logs, FAIR-VID_Decision_analysis_NoHoldout.ipynb builds the interpretable decision layer, and Process Mining Simulation.ipynb formalizes replay metrics and next-activity prediction.

  • Recover missing application IDs with bidirectional nearest-neighbor matching.
  • Normalize noisy platform events into consistent process activities.
  • Mine DFG, Petri-net, and BPMN views from the same event stream.
  • Replay traces with white-box transition logic instead of one global opaque model.

What the notebooks actually do

From raw logs to replay-ready state transitions

The journal-analysis notebook documentation exposes the preparation stages. The site now surfaces those steps so users and paper readers can see that FAIR-VID’s process model is data-engineered and inspectable, not hand-waved.

A

Recover and align log identifiers

Missing application IDs are restored using a bidirectional nearest-neighbor merge grouped by applicant. This is a concrete preprocessing step visible in the notebook-derived Python code.

B

Normalize activities

Raw platform events are mapped into lower-cardinality process activities such as passport checks, education checks, task commits, and application edits so process mining can operate on meaningful business actions.

C

Discover workflow structure

The notebooks derive directly-follows graphs, Petri nets, and BPMN views, then filter high-variance traces to separate stable mainstream paths from spaghetti remainders.

D

Transform process traces into states and junctions

Process Mining Simulation.ipynb formalizes sessionization, event abstraction, trace-prefix encoding, next-activity prediction, and replay fidelity metrics in a state-centric setting.

State-centric representation

From event logs to local transition problems

Historical logs become application states and decision junctions. At each junction, administrative variables and AI-agent outputs form a feature table, and an explicit decision rule predicts the next state.

Why not one AI model

The control problem is too heterogeneous for a single opaque predictor

  • Different stages require different evidence types and business rules.
  • Loops and exceptions create many local transition contexts.
  • Auditing is supported when each junction has explicit features and an explicit rule.
  • Historical-bias checks can be targeted to specific transitions rather than hidden across one end-to-end model.
Replay metrics

Metrics formalized in Process Mining Simulation.ipynb

Metric dimension

Action agreement

Average trace similarity between the replayed next action and the historical next action across the simulated process path.

Metric dimension

Divergence step

The first step where replay and history split, used to inspect deviations instead of relying only on one aggregate score.

Metric dimension

Outcome agreement

Whether the replay ends in the same final outcome as the historical case, even when intermediate paths are not identical.

Synthetic data role

Why the public logs are synthetic

The synthetic logs are not stand-ins for the real institutional dataset in a statistical sense. They are public data-structure and methods demos that let users and paper readers inspect event columns, trace shapes, replay logic, and state-centric orchestration without exposing personal or operational records.

  • Synthetic applicant, application, document, invoice, and administrator identifiers.
  • Synthetic or shifted timestamps.
  • Workflow traces that preserve the teaching and demonstration value of the method.
Scope note

The public demo is simpler than the server-side process twin

The broader FAIR-VID environment includes additional sources such as emails and operational data. The demo notebooks and synthetic logs are included to show methodology, not to represent every production-side integration.
